**Sandboxer 0.2.0**

Sandboxer is a Crystal shard that runs shell commands inside a platform-native sandbox with a configurable access policy. It works on macOS (via `sandbox-exec` / Seatbelt) and Linux (via `bwrap` / Bubblewrap), presenting both as a single clean API.

**Using Sandboxer as a shard**

Add it to your `shard.yml` and define a policy describing what the sandboxed process is allowed to access — paths, network, environment, working directory. Pass that policy to `Sandboxer.run` and get back a `Result` with stdout, stderr, and exit code.

```crystal
policy = Sandboxer::Policy.build do |p|
  p.read_only "/usr/share/myapp"
  p.read_write "/tmp/workspace"
  p.allow_network = false
end

result = Sandboxer.run(["python3", "script.py"], policy)
puts result.stdout
```

Policies can be loaded from JSON files, constructed in code, or composed from pre-defined presets — for example, `Sandboxer::Preset::Brew::MACOS_ARM` grants the paths needed for Homebrew-installed tools on Apple Silicon. Presets are ordinary `Policy` objects and compose via `Policy#merge`.

**Using Sandboxer as a CLI**

The `sandboxer` CLI wraps any command in a sandbox described by a JSON policy file:

```sh
sandboxer run --policy policy.json -- python3 script.py
sandboxer run --policy policy.json --add brew -- brew list
sandboxer inspect --policy policy.json --platform macos
```

`inspect` prints the native sandbox invocation (SBPL profile or bwrap flag list) without executing anything — useful for reviewing or debugging a policy before use. See the [README](./README.md) for the full CLI reference.

> The CLI is not yet distributed as a release binary. Build from source — see [DEVELOPMENT.md](./DEVELOPMENT.md).

**Platform support**

|Platform|Mechanism                           |Status                          |
|--------|------------------------------------|--------------------------------|
|macOS   |`sandbox-exec` + SBPL (Seatbelt)    |✔️ Tested (Apple Silicon + Intel)|
|Linux   |`bwrap` (Bubblewrap) user namespaces|✔️ Tested                        |

Linux requires a kernel with unprivileged user namespaces enabled. See [Known limitations](./DEVELOPMENT.md#known-limitations) in DEVELOPMENT.md for details.
